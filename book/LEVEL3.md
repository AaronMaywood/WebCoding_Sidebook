## STAGE11 ボックスモデル

CSSの重要な概念に「ボックス」があります。CSSで扱うものは全て「ボックス（４角形の箱）」です。
ボックスモデルはCSSの基本であり、レイアウトの基本でもあります。

ウェブサイトを構成するさまざまなデザイン要素は、例え素材の形が○や☆だったとしても四角のなかに収め、四角としてレイアウトされます。
レイアウト要素の全てが四角だということは、漫画のコマの扱いに似ています。漫画でも様々な形の絵が（基本的には）四角のコマに収められ、コマ割り（レイアウト）されます。

CSSでのボックスの特徴はボックスの中に子供のボックスを入れ子にできることです。そして、親のボックスを移動すればその中にある子もついてきます。その性質を利用し、親のボックスを用いてページの大規模構造を作りつつ、そのボックスの家族内で子供のボックスを配置して細部を詰めるという分割統治の考え方が可能です。

CSSはボックスの集合体であると説明しましたが、 ボックスが関わる話題は２つに分けることができます。
１つ目がボックスをレイアウトする場合の話題で、２つ目がボックスが共通に持つ内部構造についてです。

順番に説明します。この両方の特徴を押さえましょう。

１つ目の話題がレイアウト（配置）の方法です。
レイアウトには「通常のフロー（flow,流れ）」と呼ばれる既定のルールがあり、その上に`float`、`position`、`flex`、`grid`などを指定してカスタマイズしていきます。

「通常のフロー」はCSSに何も書かない時のデフォルトのレイアウトルールであり、欧文の横書きの書字方向を実装したものです。
具体的には「行は横{bdg-primary}`→`に流れ」「段落は縦{bdg-primary}`↓`に流れる」というものです。

この「通常のフロー」は既定であり日本語圏でも欧文の横書きは自然であるため、通常意識することはありません。（漫画のコマを読む順番を通常意識しないのと同様です。）
しかし「通常のフロー」はボックスが横{bdg-primary}`→`と縦{bdg-primary}`↓`に並びたがる（けっしてこれらの逆ではありません）ことを説明する重要なルールで、CSSレイアウトの基本中の基本です。

```{tip}
- アラビア語のように横書きで、行の流れが反対（左←右）の文字

`writing-mode`を用いれば「通常のフロー」をカスタマイズすることができます。
```

さて、ウェブデザインにおいて、デザイナーが扱う重要な概念の一つがレイアウト（配置）です。
素材をどのように配置するかを考えることがレイアウトを考えることに相当し、デザイナーの中で繰り返し検討される自然な考えです。

デザイナーが考えるように、CSSでも豊かなレイアウト表現ができなくてはなりません。

時代とともに、現在では豊かなレイアウト表現が可能になりましたが、最初からそうだったわけではありません。
少しCSSの歴史を振り返ってみます。

CSSが誕生した当時、ウェブは研究者同士が論文のような文章を発表しあう場でした。したがって、文書の表現能力は論文程度あれば十分とみなされていました。
この最初期に生まれたのが「通常のフロー」と`float`と`position`です。

`float`は文章の途中に挿し絵を挿入するために生まれました。**挿し絵を挿入し、その周囲にテキストが回り込むというレイアウト原則**です。

`position`は位置指定という意味で、ボックスの位置を指定する方法です。ボックスの位置指定には主に２種類あり、**画面上の位置を直接指定する絶対位置指定**`position:aboslute;`と、**一度「通常のフロー」でボックスを配置するが、そこからどのくらいずらすかを指定する相対位置指定**`position:relative`です。また位置指定されたボックスは他のボックスに重なる可能性が出てきます。 その際に**どちらを手前に表示するかという制御**(`z-indexプロパティ`)も行えます。 これは`Photoshop`のようなグラフィックツールに代表される「レイヤー」に似た画面の前後方向の多層構造を表現します。

これら`float`、`position`を裏技的に応用すると、本来の使い方から離れた様々なレイアウトを行うことができました。
このことを発見した当時のデザイナーやコーダー達が段組みなどの高度な表現を試みるようになり、ウェブの表現は当初の論文からかけはなれた、いわゆる「ウェブデザイン」の分野が切り開かれました。

この裏技の力を頼りにした灰色の時代は長く続きましたが、ウェブデザインの発展に伴って`float`に変わる本物のレイアウトの道具が必要であると広く認識されるに至りました。
そうして登場したのが`flex`や`grid`です。（あまり有名ではありませんが、`columns`もあります。）

`flexbox`は横幅いっぱいに対して、その中にどのようにボックスを配置するか（中央寄せなのか、余白を均等割にするかなど）を表現するレイアウト原則です。文章の改行と同じ用に横幅を使い切った時にボックスを改行することもできます。

`grid`は`flexbox`と似た面がたくさんありますが、`flexbox`が横方向だけを扱うのに対し、`grid`は「表（`table`）」のように縦横のマス目への配置を扱います。ウェブページの大まかなレイアウトは新聞や雑誌のように縦横のマス目を想定し、そのどの部分にコンテンツを割り付けるかを考えます（=グリッドシステム）。その考えを表現するのが`grid`というレイアウト原則です。

それまでは`float`を駆使して同等のことを行っていましたが、もはやその必要はありません。

### ボックスの話題２. ボックスの内部構造である「ボックスモデル」

ボックスのもう一つの側面は、それぞれのボックスには共通する内部構造（=**ボックスモデル**）があるというものです。

全てのボックスには玉ねぎのような階層構造があり、それによってボックス自身の大きさが決定し、また隣り合うボックス同士の余白を決めます。

ボックスの構造は内側から外側に向かって「コンテンツ（内容）」「パディング（境界線とコンテンツの間に敷かれるボックス内部の余白）」「ボーダー（境界線）」「マージン（他のボックスとの余白）」と４階層に渡って広がります。
「マージン」のみがボックスの外側の構造で、それ以外は全てボックスの内部構造です。

最も内部に位置する「コンテンツ」の大きさは、親のボックスの大きさを基準に決まり（入っている文字などの内容の大きさで決まるという例外の場合もあります）ますが、CSSの`widthプロパティ`（幅）や`heightプロパティ`（高さ）で指定することもできます。（→その場合、コンテンツがはみ出ます。はみ出たコンテンツの扱いを決める`overflow`プロパティがあります）

## 見出し 1 のボックス化・リセット CSS
{bdg-dark-line}`テキスト：P.112〜114` {bdg-primary-line}`教材：11_見出し1のボックス化・リセットCSS`

利用者の使い勝手（アクセシビリティ）の観点から、フォントサイズには単位`px`を用いず、`rem`のような相対的な単位が推奨されています（理由は下記**62.5%のトリックの解説**を参照）。そして{bdg-dark-line}`テキスト：P.112`に`font-size: 62.5%;`という表現がありますが、これは**単位`rem`を単位`px`同等の使い勝手を実現するためのトリックです**。

この設定を入れておけば、**`16px`を指定したい場所に`1.6rem`を、`40px`を指定したい場所に`4rem`（`4.0rem`）を使用できるようになります**。
（この設定をしないと、`16px`を指定したい場所には`1rem`を、`40px`を指定したい場所には`2.5rem`を指定しなければならず、直感的ではありません。）

````{tip}
**62.5%のトリックの解説**

このトリックを説明するのはなかなか厄介です。
- `Photoshop`、`figma`などのデザインツールでは単位`px`を使用してデザインします。したがってCSSでも`font-size`に`px`指定を使用したいという要求があります。
- しかしCSSでは、単位`px`を使用すると利用者の使い勝手が低下してしまう問題があります。大きいの文字で読みたいという理由でブラウザーの設定画面で既定のフォントサイズを指定したのに、CSSで指定された`px`が効いてしまうからです。このようにアクセシビリティ（使い勝手）の観点から`font-size`に単位`px`を使うことは推奨されていません。
- 単位`px`のような絶対的な表現ではなく、ブラウザーの設定値を基準にしてその何倍にするという相対的な単位`rem`を使えばこの問題を解消できます。
- `rem`はHTML文書の根本要素（ルート要素）である`HTML要素`(`<html>〜</html>タグ`)の**何倍に相当するか？**を指定する単位です。そしてルート要素の`font-size`はブラウザーに設定された既定のフォントサイズです。（何も設定をいじっていない場合は、一般的なブラウザーの場合`16px`となっています。）
	```{tip}
	`rem`は`ルート要素の文字サイズ`（`root`要素の文字サイズ`em`の略で`rem`）という意味で、この`ルート`とは`HTML要素`を指しています。
	つまり`rem`は`HTML要素`の文字サイズに対する単位です。

	なお、単位`em`と単位`rem`の違いは、単位`em`が「私の親要素のフォントサイズ」を表すのに対し、`rem`は「ルート要素のフォントサイズ(`root em`)」を表すというものです。
	% MEMO `em`はhttps://ja.wikipedia.org/wiki/Em_(%E5%8D%98%E4%BD%8D) ←ただし`em`が文字サイズ(フォントの高さ）を指すということは判然としない
	```
- CSSで単位`rem`を使用すれば、ブラウザー既定のフォントサイズを基準として、`1rem`であれば`16px`を、`1.5rem`であれば`24px`を間接的に指定したことになります。そしてブラウザーの既定フォントサイズを`32px`に変更したとしましょう。この時は`1rem`は`32px`、`1.5rem`は`48px`を指定したことになり、きちんと設定値に応じたサイズになります。このように単位`rem`を用いればアクセシビリティの問題はありません。
- しかし今度はCSS製作者がめんどうになりました。デザイナーからの指示が`40px`であった場合、いったい`何rem`を指定すればよいのでしょう？もちろんブラウザーの既定サイズは可変ですからどんな場合にも`40px`となる`何rem`はありません。したがってこの場合はほとんどの人がブラウザーの既定のフォントサイズ設定を行っていないと仮定して、ブラウザーのインストール直後の`16px`を基準にして考えることが一般に行われます。そうすれば`40px`にするためには`40➗16=2.5`で`2.5rem`を指定すればよいことになります。このような計算がいちいちめんどうであるというのが問題です。また誤りに気づくことも遅れるでしょう。
- そこで、CSS製作者側が用いるトリックが`HTML要素`に対して`font-size:62.5%;`を指定することです。これはブラウザーの既定フォントサイズが`16px`の時に、その`62.5%`であるという意味で、`16×0.625=10`はぴったり`10px`となります。
- いまや`1rem`が`10px`を指すようになり、CSS制作が楽になりました。`16px`にしたいときには`1.6rem`を、`32px`にしたければ`3.2rem`を、`40px`にしたいときには`4rem`を、というように直感的に指定できるようになったためです。
- **つまり`62.5%`は、アクセシビリティを損なわず、`rem`を`px`同等の使い勝手にするためのトリックです**。
````

### 「標準モード」とは
{bdg-dark-line}`テキスト：P.114`

ここに出てくる「標準モード（標準ボックスモデル）」という用語を補足します。
「標準モード（標準ボックスモデル）」に対する言葉は「代替ボックスモデル」です。

この両者は、次の節に登場する`box-sizing`プロパティで切り替えることができます。
`box-sizing: border-box`を設定したときが「代替ボックスモデル」で、設定しない時（初期値の`box-sizing: content-box;`の時）が「標準ボックスモデル」です。

「標準ボックスモデル」と「代替ボックスモデル」の違いは、幅`width`（及び高さ`height`）の解釈の違いです。
% {{TODO}}ここにwidth解釈の違いの絵が欲しい

幅`width:100px`を指定した場合で説明します。
標準ボックスモデルでは、幅`width:100px`は、「内容（コンテンツ）」の幅のみを指します。
このボックスに`padding: 50px`、`border: 5px solid black`（太さ5pxの境界線）が指定されている場合は、ボックス全体の幅は`100+50+5=155px`となります。

それに対し、代替ボックスモデルでは幅`width:100px`が「内容（コンテンツ）」と「パディング」「ボーダー」の３つを併せたものを指します。
したがってボックス全体の幅は`100px`のままです。

さて、この両方の幅の解釈ですが、レイアウトする時に直感的に分かりやすいのは「代替ボックスモデル」の方です。
「代替ボックスモデル」はボックスの大きさを把握するのが簡単です。
このため、`box-sizing: border-box`をリセットCSSに含めることも行われます。

### 「幅」と「高さ」を「border」と「padding」を含めた値にする ─ box-sizing: border-box
{bdg-dark-line}`テキスト：P.114`

## margin や padding プロパティの一括設定（ショートハンド設定）
 {bdg-dark-line}`テキスト：P.115`

ショートハンドは短略形という意味で、複数のプロパティを一括で指定できる短縮表現です。
```{code} css
例)
margin-top: 10px;
margin-right: 10px;
margin-bottom: 10px;
margin-left: 10px;
↓（上記４つのプロパティを一行で表した短略形） 
margin: 10px;
```

`margin`と`padding`の短略形は同じ書き方なので、`margin`を覚えれば`padding`も覚えたことになります。

## ボーダーの作例
{bdg-dark-line}`テキスト：P.116` {bdg-primary-line}`教材：12_ボーダー`

`margin`や`padding`はショートハンドだと紹介しましたが、ここで説明する`border`もまたショートハンドです。
`border`は`border-width`、`border-style`、`border-color`の３つのプロパティのショートハンドです。
```{code}
border-width: 2px;
border-style: solid;
border-color: green;
↓（短略形）
border: 2px solid green;
```

````{tip}
CSSではショートハンドはありふれたもので、`background`、`font`、`box-shadow`など、ショートハンドは至るところにあります。
ショートハンドを利用する際のお約束として、ショートハンドを具体的なプロパティと共に使用する時の注意点を押さえておきましょう。

マージンを例に説明します。マージンを上部のみ`50px`とし、その他は`0px`にしたいとします。
このように書いてはなりません。
```{code}
body {
	margin-top: 50px;
	margin: 0;
}
```
これは、`margin: 0;`が後に書いてあり、かつ`margin: 0;`は`margin-top: 0;`の意味も含むため、上書きされてしまうからです。

したがって、このようにして下さい。
```{code}
body {
	margin: 0;
	margin-top: 50px;
}
```
こうすれば`margin: 0;`の指定を`margin-top: 50px`が上書きすることになります。

これは４方向をそれぞれ指定する短略形`margin: 50px 0 0 0;`を使用しても同じ意味になります。
しかし、マージン以外の短略形全てにこのルールが適用されるため、短略形と細かな指定を共に書く時にはこのルールを思い出して下さい。
````

## ボックスの外部余白（margin）の相殺（そうさい）について
 {bdg-dark-line}`テキスト：P.137〜138` {bdg-primary-line}`教材：23_相殺について`

隣り合ったボックス同士にマージンが設定されている場合の扱いに関する話題です。

**隣り合ったボックスのマージンの量は、両方のマージンの足し算ではなくどちらか長い方が採用されるというルールがあり、これをマージンの相殺と言います**。

**相殺**があるのは`margin`だけです。`border`、`padding`にはありません。

相殺という言葉が難しいのでたとえ話で説明します。

一軒家があったとします。その一軒家の持ち主Aは、隣の家との間隔（マージン）を`5m`は欲しいと思っていました。
そこへ隣の家を建てることになり、その隣家の持ち主Bも隣の家との間隔を最低`5m`にしたいと思っていました。
この時、二軒の家の間を**5m**にすれば両者の希望を満たせます。
両者の希望は`5m`あればよいというものであって、数値を足して`5m+5m=10m`もの過剰な長さを確保する必要はありません。

数値だけをおいかければ`5m+5m=10m`とも思ってしまえますが、「余白」の性質から`5m`で十分だとなります。
この性質を（数値上は相殺されているように見えるため）「マージンの相殺」と呼んでいます。

**相殺は両者のマージンのうち長い方を採用する**という意味になります。
もし隣家の持ち主Bが`7m`の間隔が欲しいと思った場合には、両者の要求する間隔(Aは`5m`を希望, Bは`7m`を希望)のうち、長い方(`7m`)を選択します。

ボックスモデルを構成する「マージン」「ボーダー」「パディング」「内容（コンテンツ）」のうち、相殺があるのは隣家との間隔を決めるマージンだけです。
その他の「ボーダー」「パディング」「内容（コンテンツ）」は家の内部の話であり、隣接するボックス同士の関係である相殺は関与しません。
